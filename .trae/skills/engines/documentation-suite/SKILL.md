## 📋 Standard Operating Procedure

### Documentation Development Pipeline

#### Phase 1: Discovery & Audience

1. **Requirements Gathering**
   ```
   ▢ Who is the primary audience?
   ▢ What do they already know?
   ▢ What are they trying to accomplish?
   ▢ What frustrates them about existing docs?
   ▢ What format do they prefer?
   ▢ Success metrics for good documentation
   ```

2. **Information Architecture**
   ```
   ▢ Table of contents outline
   ├── Getting Started
   ├── Tutorials
   ├── How-To Guides
   ├── Reference
   ├── Explanation / Background
   └── Examples
   ▢ Progressive complexity curve
   ▢ Cross-reference mapping
   ```

#### Phase 2: Content Creation

3. **Reference Documentation**
   ```
   ▢ Every public API documented
   ▢ Parameters: name, type, required, default
   ▢ Return values with types
   ▢ Exceptions and error conditions
   ▢ Complete, runnable examples
   ▢ Edge cases and caveats
   ```

4. **Guides and Tutorials**
   ```
   ▢ Learning objectives stated upfront
   ▢ Prerequisites clearly listed
   ▢ Step-by-step, numbered instructions
   ▢ Expected output at each step
   ▢ Checkpoints for verification
   ▢ Common pitfalls and how to avoid
   ```

5. **Architecture and Explanation**
   ```
   ▢ System context diagram
   ▢ Component relationships
   ▢ Data flow description
   └── A flows to B which triggers C
   ▢ Design decisions and tradeoffs
   ▢ Why we did it this way, not that way
   ▢ Alternatives considered and rejected
   ```

6. **Academic Writing**
   ```
   ┌─────────────────────────────┐
   │ Abstract                    │
   │ Introduction                │
   │ Related Work                │
   │ Methodology                 │
   │ Implementation              │
   │ Evaluation / Results        │
   │ Discussion                  │
   │ Conclusion                  │
   │ References                  │
   └─────────────────────────────┘
   ```

#### Phase 3: Quality & Delivery

7. **Quality Assurance**
   ```
   ▢ Technical accuracy review
   ▢ Clarity and readability check
   ▢ Spelling and grammar
   └── Consistent terminology
   ▢ All examples tested working
   ▢ All links verified
   ▢ Accessibility compliance
   ```

8. **Publication & Maintenance**
   ```
   ▢ Docs-as-code structure
   ▢ Versioning strategy
   └── v1, v2, etc. parallel docs
   ▢ Changelog and deprecation notices
   ▢ Feedback collection mechanism
   ▢ Regular update schedule
   ```

## 📚 Embedded Knowledge Base

### Divio Documentation System

The golden standard, always follow:

| Type | Purpose | Example |
|------|---------|---------|
| **Tutorials** | Learning-oriented, takes you by hand | "Build your first X in 30 minutes" |
| **How-To Guides** | Problem-oriented, goal-directed | "How to do Y specific task" |
| **Reference** | Information-oriented, technical | "API: function X(params)" |
| **Explanation** | Understanding-oriented | "Why we designed it this way" |

### Academic Paper Structure Checklist

✅ **Abstract**: 150-250 words, covers motivation, approach, results, implications

✅ **Introduction**: Context, problem statement, contributions, paper structure

✅ **Related Work**: Fair comparison to prior work, positioning your contribution

✅ **Methodology**: Reproducible, precise, justifies design choices

✅ **Implementation**: Details, challenges, engineering tradeoffs

✅ **Evaluation**: Baselines, metrics, statistical significance, threats to validity

✅ **Discussion**: What worked, what didn't, lessons learned, limitations

✅ **Conclusion**: Summary, future work, broader impact

✅ **References**: Complete, consistent, properly formatted citations

### Technical Writing Principles

1. **Write for the reader, not the writer**
   > You already know how it works. They don't.

2. **Show, don't just tell**
   > Working code example > 1000 words of explanation.

3. **Assume the reader makes mistakes**
   > Tell them what error messages to expect and how to fix.

4. **Consistent terminology**
   > If you call it a "widget" on page 1, don't call it a "gadget" on page 5.

5. **Anticipate questions**
   > Why would anyone do this? When should I NOT do this?

6. **No "simply" or "just"**
   > These words are insults. If it were simple, they wouldn't be reading the docs.
